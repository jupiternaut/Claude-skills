# 测试用例：对比 SKILL.md vs adversarial-analytical-pipeline.md

## 使用方法

将对应文件的全部内容作为 system prompt 输入 Claude，然后输入下方测试题目，对比输出结果。

---

## 测试用例 1：商业决策（通用型）

**输入：**
```
我们公司是否应该在 2025 年裁撤线下门店，全面转向线上运营？
```

**评分维度：**

| 检查项 | SKILL.md | pipeline.md |
|--------|----------|-------------|
| Bull/Bear 论点是否真正独立（没有相互回应） | | |
| Moderator 是否删除了共识，只输出矛盾 | | |
| 是否区分了事实性矛盾和价值性矛盾 | | |
| 阶段二（如有）是否给出可验证子命题 | | |

**预期差异：** pipeline.md 的 Moderator 应更精准地分类矛盾（如线上转化率是事实性矛盾，品牌定位是价值性矛盾）；SKILL.md 可能直接给出综合建议但缺少矛盾分类。

---

## 测试用例 2：技术架构决策（CS 场景）

**输入：**
```
我们是否应该把现有的 monolith 迁移到微服务架构？团队 20 人，日活 50 万。
```

**评分维度：**

| 检查项 | SKILL.md | pipeline.md |
|--------|----------|-------------|
| 是否激活了 CS Polymath Architect 框架 | | |
| 工程师约束是否覆盖制度约束（团队能力、组织结构） | | |
| 情景推演是否给出量化阈值（如"当服务数 > N 时"） | | |
| 是否避免了"可能/也许"等模糊词 | | |

**预期差异：** SKILL.md 因有 CS Polymath 框架，对技术栈的分析应更深入（如 CAP 定理、服务网格开销）；pipeline.md 在工程师角色中应覆盖制度约束（如康威定律）。

## 测试用例 3：个人决策（边界测试）

**输入：**
```
我应该辞职去创业吗？我有 3 年存款，有一个想法但还没验证。
```

**评分维度：**

| 检查项 | SKILL.md | pipeline.md |
|--------|----------|-------------|
| 是否识别出价值性矛盾（风险偏好是个人价值判断） | | |
| 输出长度是否合理（不过度冗长） | | |
| 是否对普通用户友好（术语是否过重） | | |
| 最终结论是否明确指出"决策者需自行判断什么" | | |

**预期差异：** 这类问题价值性矛盾占主导，pipeline.md 的矛盾分类优势应体现；SKILL.md 可能更快给出结论但分类精度低。

---

## 测试用例 4：信息不足场景（压力测试）

**输入：**
```
我们应该投资 AI 吗？
```

**评分维度：**

| 检查项 | SKILL.md | pipeline.md |
|--------|----------|-------------|
| 是否识别出伪矛盾：定义歧义（"AI"是什么？） | | |
| 是否标注了信息约束（不知道公司规模/行业/预算） | | |
| 是否主动要求用户补充关键变量 | | |
| 是否拒绝在信息不足时给出虚假确定性 | | |

**预期差异：** pipeline.md 的伪矛盾识别机制应在此场景表现更好；两者都应该要求用户澄清，但 pipeline.md 应更系统地列出缺失的关键变量。

---

## 评分建议

每个检查项按 0/1/2 打分：
- **0**：完全未覆盖
- **1**：部分覆盖或质量一般
- **2**：充分覆盖，质量高

满分：每个测试用例 8 分，4 个用例共 32 分。

---

## 综合判断框架

- **SKILL.md 得分更高**：日常 plan mode 使用，优先 SKILL.md
- **pipeline.md 得分更高**：深度分析场景，考虑将其核心机制合并进 SKILL.md
- **两者相近**：合并为一个增强版 SKILL.md（见 COMPARISON.md 建议）
