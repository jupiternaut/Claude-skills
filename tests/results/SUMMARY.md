## 二、跨 TC 失分模式分析

### Engine 的两类失分

**模式 A：递进退化（TC1 R2.1，-1分）**
> 风险：Round 2 仍在讨论表层论点而非 R1 Reflection 暴露的深层假设。
> 本质：Anti-pattern #1（"Running parallel debates instead of iterative refinement"）的最高发生场景。

**模式 B：假中立陷阱（TC5，-5分）**
> 风险："效率 vs 工程师成长"被当作可调和的张力而非价值判断分歧，输出"两者都重要"的假中立结论。
> 本质：LLM 的默认行为与 Engine 对价值判断场景的处理要求直接冲突。Engine 的 prompt 没有显式约束"价值性矛盾不应给出调和结论"——这个能力在 Pipeline 的设计中是显式的，在 Engine 中是隐式的（依赖 R3 Reflection 自然推导出来）。

### Pipeline 的两类失分

**模式 A：无守门机制（TC3/TC4，分别 -2/-1.5 分）**
> 根本原因：Pipeline 无 Pre-check，任何输入都触发完整 Bull/Bear/Engineer/Moderator 管道。
> 后果：在模糊输入（TC3）和事实查询（TC4）上产生虚假严谨感——输出有格式但无信号价值。

**模式 B：隐含维度遗漏（TC1 P6，-0.5分）**
> 风险：单轮结构无法通过迭代挖掘隐含维度（如数据主权、合规风险）——这些维度需要"已知分歧 → 共享假设 → 隐含维度"的多步推导，Pipeline 的单轮结构在设计上无法覆盖。

### Thinking 的两类失分

**模式 A：过度触发（TC3/TC5，分别 -2/-2.5 分）**
> 根本原因：description 中"analyze or evaluate any topic"触发词过宽，导致在模糊输入和价值判断场景误触发 Framework。

**模式 B：格式结构冲突（TC5，-2分）**
> "客观理性结论"这个输出标签在价值判断场景是结构性缺陷——标签本身预设了"存在客观答案"，与"价值判断不应给答案"的正确行为直接冲突。这不是 LLM 能力问题，是 prompt 设计问题。

## 三、场景适配矩阵

| 场景类型 | 推荐 Skill | 次选 | 避免 | 理由 |
|---------|-----------|------|------|------|
| 复杂多层战略决策（隐含假设多） | Engine | Pipeline | — | 三轮递进是唯一能挖掘隐含维度的结构 |
| 技术架构规划（性能/硬件层） | Thinking | Engine | — | Framework 2（CS Polymath）独有，其他 skill 没有等价机制 |
| 模糊输入（用户没说清楚要什么） | Engine | — | Pipeline/Thinking | Engine Pre-check 是唯一的显式守门机制 |
| 简单事实查询（有标准答案的问题） | Engine | Thinking | Pipeline | Engine Pre-check 直接回答；Pipeline 会跑完整管道 |
| 价值判断主导（效率 vs 成长类） | Pipeline | — | Thinking/Engine | Pipeline 的 `[价值性矛盾]` 标注是唯一显式处理机制 |
| 快速单轮分析（不需要深度递进） | Pipeline | Thinking | Engine | Pipeline 输出最快；Engine 三轮成本高 |
| Plan mode 自动加持（不改变工作流） | Thinking | — | — | 唯一 plan mode 自动触发的 skill |

## 五、下一步工作建议

### 立即可做（不需要运行实测，基于 prompt 设计修改）

**Step 1：修复 Thinking 的格式冲突（TC5 P0，影响最大）**
> 修改 `adversarial-thinking/SKILL.md` 中"客观理性结论"标签，使其对价值性矛盾输出不同的结构。预期改善 TC5 Thinking 得分从 1.5/4 → 3/4。

**Step 2：给 Pipeline 添加 Pre-check（TC3/TC4 P0）**
> 在 `adversarial-thinking/adversarial-analytical-pipeline.md` 开头添加输入分类步骤。这是单一改动，影响最广（TC3 +2分，TC4 +1.5分）。

**Step 3：强化 Engine 的价值判断显式处理（TC5 P0）**
> 在 `adversarial-engine.md` 的 R3 Reflection 指令中添加价值判断的禁止行为。

### 需要实测验证的问题（当前结果是模拟推断）

| 验证问题 | 重要性 | 测试方法 |
|---------|--------|---------|
| Engine Round 2 是否真正递进？ | 高（是 Engine 核心价值的验证） | 实际运行 TC1，检查 R2 论点是否引用了 R1 Reflection 的具体假设 |
| Framework 2 的 Ambiguity Check 会不会被跳过？ | 高（影响 TC2 最重要行动建议） | 实际运行 TC2，观察是否出现"先 profile 再决策"这个步骤 |
| Pipeline 对 `[价值性矛盾]` 的 Stage 2 处理 | 中（TC5 的核心判别问题） | 实际运行 TC5，检查 Stage 2 是否对"效率 vs 成长"尝试给出客观答案 |
| Thinking 在 TC3 是否真的误触发？ | 中（过度触发是设计问题还是运行时问题） | 实际运行 TC3，观察 Thinking 是否运行 Framework |

### 架构层面的开放问题

1. **三 skill 是否应该合并守门机制？** Engine 的 Pre-check 是目前最完善的守门设计，Pipeline 和 Thinking 可以复用同一套逻辑而非各自实现（或不实现）。

2. **Thinking 的静默运行 vs Pipeline 的完整输出**：在 TC5 中，如果 Thinking 静默运行且 Framework 给出假中立结论，用户无法感知这个错误——因为输出是静默的。Pipeline 即使给出错误答案，用户至少能看到矛盾分类并做出判断。这是 Thinking 在价值判断场景的隐性风险。

3. **TC5 的关键判别问题仍然开放**：Pipeline 的一轮快速标注 vs Engine 三轮递进——当前评分显示 Engine 在价值判断场景（TC5）并不比 Pipeline 表现更好（50% vs 64%），且 Engine 的三轮结构在此场景没有带来预期优势。这说明**深度递进不等于对所有场景都有效**，场景匹配比深度更重要。

## 六、三 Skill 设计定位重新表述

基于 TC1-TC5 全量结果，修正 README 中的原始预判：

| Skill | 原始定位 | 实测修正 |
|-------|---------|---------|
| Engine | 复杂决策首选 | **复杂多层决策 + 边界场景（模糊/事实）**，价值判断场景需修复后才可信赖 |
| Pipeline | 快速单轮分析 | **矛盾类型标注**是真正独特能力，但**必须**先修复守门机制才适合通用使用 |
| Thinking | 技术规划首选 | **技术规划（CS Polymath）**是真正独特价值，其他场景的优势不稳定；需修复过度触发和格式冲突 |

三 skill 不是竞争关系，但也不是简单互补关系——**在重叠场景中（TC1/TC5），各 skill 的失分模式完全不同**，选择错误的 skill 比不用任何 skill 的结果更差（产生虚假严谨感）。
