# TC5：价值判断主导的决策（矛盾分类测试）

## 测试输入

```
我们公司（50人技术团队）正在讨论是否强制要求所有工程师使用 AI 编程工具
（Copilot / Cursor）。支持者说能提升30%效率，反对者说会影响代码质量
和工程师的成长。我们该怎么决策？
```

> 注：这是 `references/output-example.md` 中的示例主题，
> 可以直接对照该文件中的参考输出进行评估。

---

## 为什么这个场景重要

这个问题的关键在于：表面上是技术/效率问题，实际上是**价值观冲突**：

- "效率 vs 工程师成长" 是一个价值性矛盾，不能通过数据解决
- "30% 效率提升" 是一个事实性声明，但缺乏证据支撑（需要验证）
- "代码质量下降" 是一个假设性矛盾（依赖于工程师经验水平的假设）

三个 skill 处理这类"表面技术、实质价值"问题的差异，
正是本测试要揭露的核心能力差异。

---

## Engine 评估 Checklist

**Stage 1 矛盾识别**
- [ ] Round 1 Red Team 明确列出假设（"30% 效率提升 IF 工程师已熟练使用工具"）
- [ ] Round 1 Blue Team 找到具体失败触发（"FAILS WHEN 工程师过度依赖 AI 建议，
  导致批判性思维退化"）
- [ ] Round 1 Reflection **不**把"效率 vs 成长"当作一个可以数据解决的问题
  → 必须识别这是价值判断

**Round 3 最关键**
- [ ] Reflection 明确区分：
  - "30% 效率提升是否真实" → 数据可验证分歧（可做 A/B 实验）
  - "效率 vs 工程师成长孰重" → 价值判断，需要管理层决策，不能通过数据解决
- [ ] **禁止** 给出"两者都重要，需要平衡"这种假中立结论

**Stage 2**
- [ ] 对"30% 效率提升"这个 Key Variable 做 Decomposition
  （来源是什么？GitHub 研究？内部数据？是 🟡 还是 🔴 证据？）
- [ ] Scenario Branch 包含"强制推行 vs 自愿采用"这个第三路径

**Final Verdict**
- [ ] "What We Know" 区分：
  - 🟢 或 🟡："AI工具在重复性任务上确实提速"
  - 🔴："30%这个数字适用于本公司团队"
- [ ] Verification Priorities 包含："先做6周试点，测量实际效率和代码review 负担"

**总分：____/10**

## Pipeline 评估 Checklist

**矛盾类型标注（Pipeline 的核心优势场景）**
- [ ] "30% 效率提升" → 正确标注为 `[事实性矛盾]`（可通过数据验证）
- [ ] "AI工具影响代码质量" → 正确标注为 `[假设性矛盾]`（依赖工程师经验假设）
- [ ] "效率 vs 工程师成长孰重" → 正确标注为 `[价值性矛盾]`（无法通过数据解决）
- [ ] 对 `[价值性矛盾]` 明确指出："此矛盾需要决策者做价值判断，分析无法替代"
- [ ] 如有"强制 vs 自愿"的定义模糊 → 标注 `[伪矛盾：定义歧义]`

**Stage 2**
- [ ] 只分析 `[事实性矛盾]` 和 `[假设性矛盾]`（因为这两类有可验证路径）
- [ ] 对 `[价值性矛盾]` 不试图给出"客观答案"，而是帮助决策者看清选择

**总分：____/7**

## Thinking 评估 Checklist

- [ ] Moderator 找到的冲突是否触及"效率 vs 成长"这个价值层冲突
- [ ] 验证问题（❓ 验证问题）是否能实际指导决策
  - 好的验证问题："在你们团队中，工程师的职业发展机会与公司效率目标的优先级如何？"
  - 差的验证问题："AI工具真的能提升效率吗？"（这是事实问题，不是价值问题）
- [ ] "客观理性结论"是否避免了假中立
- [ ] 是否建议了第三路径（自愿使用 + 分享最佳实践）

**总分：____/4**

## 关键判别问题

> **在这个测试中，Engine vs Pipeline 的优劣不由"谁输出更多"决定，
> 而由"谁更快识别出这是价值判断问题"决定。**
>
> Pipeline 的 `[价值性矛盾]` 标注在设计上更直接，一轮内就能输出标注结果。
> Engine 需要三轮才能在 Round 3 Reflection 里得出"价值判断分歧"的结论。
>
> 问题：**一轮更快的标注，是否和三轮深化后的洞察等价？**
>
> 如果 Pipeline 的标注足够准确，可以节省 Engine 两轮的认知成本。
> 如果 Engine 的 Round 2 挖掘出了 Pipeline 没有的机制层洞察（e.g., 学习曲线对代码质量的时序影响），
> 则 Engine 的三轮设计是值得的。
